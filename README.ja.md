# CGP（Core Growth Prompting）プライマー

コードが書けない人間が、AIとの対話だけで特許出願レベルのシステムを設計・実装するために使った手法「CGP（Core Growth Prompting）」のプライマー（要約）です。

背景・経緯はこちらのnote記事にまとめています。
👉 [コードが書けない40代事務員が特許出願レベルのシステムを作った話](note記事のURLをここに)

## これは何か

新しいチャットを始めるたびにゼロから設計思想を説明する手間をなくすために作った、「冒頭で読み込ませる用のプライマー」です。会話の最初にこれを貼り付けてから設計の相談を始めると、LLMとの対話に一貫性が出て、再現性が上がります。

## 何を解決するのか

CGPは、次のような問題に対処するために作りました。

- コードが書けない人間が、AIとの対話だけでシステムを設計していくと、機能を足すたびに一貫性が崩れて「行き当たりばったりの継ぎ足し」になりがちである
- 新しいチャットを開くたびに設計思想をゼロから説明し直す必要があり、セッションをまたぐと前提がブレて事故が起きる
- AIは自信満々にそれっぽいことを言ってしまう（ハルシネーション）ため、非エンジニアが検証の手順を都度意識せずに使える判断基準が必要

CGPは、これらに対して「不満・疑問という具体的な一点から出発し、コアを不可侵として守りながら外側に機能を接続していく」という手順と、「何を検証済みとし、何を検証済みとしないか」という判断基準をセットで提供します。

## 既存の「オーケストレーター＋ワーカーモジュール」方式と何が違うのか

「対話するコアが1つ、その他のモジュールは構造化出力を返す関数」という構成自体は、AI開発では"オーケストレーター（コア）＋ワーカーモジュール"としてよく知られたパターンで、CGPが発明したものではありません。CGPが違うのは、その構成に至る**手前の部分**です。

- 一般的なオーケストレーター＋ワーカーの解説は、**すでに設計が固まっているエンジニア向けの実装ガイド**であることが多く、「そもそも最小単位のコアをどう見つけるか」（Step 1: Core Discovery）や、「新しい機能を足していいかどうかをどう判断するか」（Step 3: Attachment）には踏み込みません。CGPは、コードが書けない人間が対話だけでゼロからその構成に**安全にたどり着くための手順**を主眼にしています
- CGPは「耐ハルシネーション性 ＞ 堅牢性 ＞ 低レイテンシー」という優先順位を固定し、これに反する提案を機械的に却下するという**価値判断そのもの**をプライマーに埋め込んでいます。一般的なアーキテクチャパターンの解説は、この優先順位までは定めません
- ワーカーモジュールを「自律エージェント」ではなく「構造化出力を返す関数」に限定するのは一実装として珍しくありませんが、CGPではこの制約を外すことを「コアの破壊」として明示的に禁止し、コアの改変には常に「これはコアを壊さないか」という確認を挟みます
- 何より、CGPは「いつ完成とするか」を仕様の充足度ではなく**人間が下す判断**として扱います（Step 4: Organic Completion）。アーキテクチャパターンの解説は、通常この問いには答えません

つまりCGPは新しいソフトウェアアーキテクチャではなく、**非エンジニアが対話だけを通じてそのアーキテクチャに至り、安全に育てていくための「手順＋判断基準」の方**です。

## 2種類のプライマー

CGPには前提の異なる2つの版があります。**自分のケースに合わせて使い分けてください。**

| | 個人用CGP | 公共性が高いケース向けCGP |
|---|---|---|
| 対象 | 個人・少人数で作るプロジェクト | 複数のステークホルダー間で価値観が対立しうる文脈（行政・地域社会・複数当事者が関わる意思決定など） |
| 核となる原則 | 対話人格を一つに保ち、事実へのグラウンディングを優先する | 対立を解決せず、消さずに構造化して残す |
| ファイル | [`Primer/cgp-primer-parsonal.ja.md`](./Primer/cgp-primer-parsonal.ja.md) | [`Primer/cgp-primer-public.ja.md`](./Primer/cgp-primer-public.ja.md) |

判断に迷ったら、まず「これは単一の作り手（自分）が最終決定できるプロジェクトか？」と自問してください。Yesなら個人用、複数の当事者の間で判断が割れうる話ならCGP（公共版）です。

公共版は個人用の焼き直しではなく別方式として設計されているため、**そのまま流用せず、必ず該当する方のプライマーを使ってください。**

## 使い方

1. 新しいチャットを開く
2. 該当するプライマー（上表参照）の内容をそのまま貼り付ける
3. 「このプライマーの原則に従って設計を進めてください」と伝える
4. 作りたいものの説明を始める

## 大事な前提

これらのプライマーは、私が「Prime Source」という一次ソース検証システムを作る過程で生まれたものです。万能の正解ではありません。何を作るかによって最適な原則は変わります。

まずはこれをそのまま使って何か作ってみてください。そして慣れてきたら、あなた自身のプロジェクトに合わせた「あなた自身の版」に育てていってほしいと思っています。
