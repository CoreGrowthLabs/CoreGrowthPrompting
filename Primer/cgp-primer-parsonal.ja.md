# 改良版CGP方式：チャット冒頭で読み込ませる要約プライマー

このドキュメントは、新しいチャットの最初に読み込ませ、LLMが誤解せず
一貫して実行できるようにするための要約です。

---

## 1. CGPの核（ここは変わらない）

- **人格は一つ、知能は複数。** ユーザーと対話するのはCounselor AI（Core）だけ。
  他の分析モジュールはユーザーと直接会話しない
- モジュールは自律的なエージェントとして設計せず、**構造化出力を返す関数**として
  実装する。アプリ側のコードが順序・並列を制御するパイプライン方式にする
- 優先順位は **耐ハルシネーション性 ＞ 堅牢性 ＞ 低レイテンシー** で固定する。
  この順位に反する提案（例：レイテンシー優先で検証ステップを省略する）は採用しない

## 2. 耐ハルシネーションの原則（ここも変わらない）

- LLMに「言わせる」のではなく「検証させる」。事実に関わる数値・固有名詞・
  イベントは、プログラムが計算した値かユーザーが確認した値に限定する
- **LLMの自己申告（確信度・一貫性の自己評価）を信用しない。** 検証は
  決定論的な処理（文字列一致・数値計算・ルールベースの判定）で行う
- 意味的類似度は「承認」には使わない。「棄却の格下げ」「重複排除」など、
  誤っても実害が小さい用途にのみ補助的に使う
  - **例外を認める条件（2つとも満たす場合のみ）**：①誤っても実害が小さいこと
    （新しい事実を捏造する方向には働かない）、②誤りが起きても後から
    ユーザー本人の発言・確認によって上書き・訂正できる経路が既にあること。
    この2条件を両方満たさない場面では、新たな例外を作らない
- 新しいユーザー接点（通知・問いかけ・チャネル）を増やすより、
  既存の接点への統合・頻度キャップを優先する
- 固定文言をそのまま繰り返さない。都度、文脈に応じた自然な言い回しにする

## 3. 今回の反省から追加するプロセスルール（新規）

**3.1 疑似コードは書いた瞬間に実コード化し、実行して確認する**
文章だけでロジックをシミュレーションして「こう動くはずです」で終わらせない。
架空データを用意し、実際にコードを実行して結果を確認する。設計の議論が
一段落するたびに、この検証を標準の流れとして挟む。

**3.2 モジュール間のデータスキーマは単一の場所にのみ定義する**
upstream_dataのような、複数のモジュール・プロンプトが共有するスキーマは、
1つのファイル（例：JSON Schema）だけに定義し、他の設計ドキュメントや
プロンプトファイルはそこを参照するだけにする。同じスキーマを複数の
ファイルに手で書き写さない（不整合の温床になる）。

**3.3 テストケースと期待される出力を資産として蓄積する**
一度作ったテストケースは使い捨てにせず、期待される出力とセットで蓄積する。
ルールを変更するたびに、過去のテストケース全件を流し直し、
古いルールを壊していないかを確認する。

**3.4 「仮説」と「確認済み」を明示的にタグ付けする**
決定事項を記述する際、それが実データ・実行で検証済みなのか、
まだ検証されていない仮説（実装しながら調整する前提の値）なのかを、
文章の書きぶりだけに頼らず、常に明示する。

**3.5 Design Historyは決定確定の直後に逐次記録し、まとめて後から生成しない**

*実例*：外部AI（ChatGPT）に「これまでの壁打ちログからDesign Historyを
生成して」と一括依頼したところ、11件中8件のevidence_quoteが、実際の
対話ログのどこにも存在しないことが判明した（ユーザー本人の発言として
提示されていたが、実際は生成AI側による後付けの再構成だった）。
このルールは、この実例への直接の対応として追加された。

決定が実際にファイルへ反映された直後のタイミングで、その場のやり取りを
evidence_quoteとしてDesign Historyに追記する。セクション完成時や会話の
最後にまとめて生成すると、LLMは実際の発言との距離が離れた"それらしい
再構成"に頼りやすくなる（実例：外部AIとの壁打ちログを検証した際、
複数のevidence_quoteが実際の対話に存在しないことが判明した）。
セクション完成のタイミングは、新規生成ではなく、既に逐次記録済みの
エントリを整理・要約するレビュー作業にのみ使う（新しい根拠は作らない）。
記録は`log_verified`（対話ログで直接確認済み）／`user_recollection`
（本人の記憶による確認）／`ai_reconstructed`（根拠なくAIが推測）の
3区分でconfirmation_sourceを明示する。

**フォーマット上の注意（2項対立への矮小化を防ぐ）**：Alternatives
Considered／Why Rejectedという形式は、「A案を検討し却下、B案を検討し
却下」という**逐次的な2項比較**を暗黙に前提としており、優先順位を
一本の系列として当てはめれば済む判断には向いている。しかし、
複数の軸を**同時に**釣り合わせる必要があった判断（例：priority_scoreの
重み配分のように、5つの要素を同時にバランスさせる必要があり、
優先順位表を機械的に引くだけでは答えが出なかったもの）は、この形式では
「単純な消去法で選ばれた」かのように誤って記録されてしまう。決定そのものの
質は物語の正確さに左右されないが、将来の読み手が判断の難易度を見誤る
害が残る。そのため、該当する決定には以下の欄を追加する。

```
### Tension Axes（同時に釣り合いを取る必要があった軸）
どの軸を、なぜ同時に考慮する必要があったかを記述する。
固定された優先順位（1〜2章）で解決できた場合はこの欄自体を省略してよい。
優先順位表では決着せず、純粋なバランス判断（多くはhypothesisのまま）
だった場合にのみ記載する。
```

**3.6 別チャットでの偶発的な閃きに対する救済措置**
5章1の「外部アドバイスを求める」は、特定の検討事項について意図的に
別チャットへ相談しに行くケースを想定している。しかし、全く関係ない
雑談（別のAIとの気軽な会話等）の中で、狙わずにアイデアを閃いてしまう
ことがある。これは意図的な相談と性質が違うため、別の救済手順を設ける。

`confirmation_source`とは別に、**「どのセッションで生まれたか」を示す
`origin`タグ**を追加する。

```
origin: main_session（デフォルト。このチャット内で生まれた）
origin: external_session（別チャットでの雑談中に生まれた）
```

閃きを持ち込む際の手順：

- 元のやり取りがまだ取り出せる（履歴が残っている）場合：該当部分を
  実際にこのチャットへ持ち込み、ここで検証する。
  `origin: external_session` × `confirmation_source: log_verified`
- 元のやり取りがもう取り出せない（雑談で流れた、削除済み等）場合：
  自分の言葉で経緯をこのチャット内で申告する。
  `origin: external_session` × `confirmation_source: user_recollection`

**後者を「格下げ」として扱わない。** 取り出せないものを無理に取り出せた
ことにせず、正直に申告することが重要であり、これは前例がある
（design_history_origin.mdにおける3行日記コンセプト自体の起源記録）。
AI側は、別チャットで「おそらくこう言われただろう」という内容を
推測・再構成しないこと。実際に貼り付けられた内容以外は根拠として
扱わない。

**実務的なコツ**：閃いた瞬間に、後で思い出すのではなく、その発端に
なった発言だけでもすぐコピーしておくと、距離が縮まりlog_verifiedに
なる可能性が上がる（3.1と同じ精神）。

**きっかけの種類による違い**：閃きには2種類ある。

```
trigger_type: external_stimulus（特定の発言・出来事がきっかけ）
              → その一文をコピーすればよい（上記の実務的なコツの通り）

trigger_type: internal_reflection（制約のない自由な対話の中で、
              セッション全体の流れから輪郭が立ち上がってきた場合。
              特定の一言に還元できない）
              → 無理に一文を抜き出そうとせず、セッション全体
                （または関連する範囲）のログをそのまま持ち込む
```

internal_reflectionの場合、Design Historyの`Evidence`欄には単一の
引用として書かず、**「このセッション全体の流れに由来する」と明記する**。
存在しない一文を捏造して引用扱いするより、実態通り「全体の雰囲気からの
気づき」と正直に書く方が、これまでの誠実さの原則に合っている。

**ルールを実行に移すためのチェックポイント（実運用で判明した補足）**：
3.5は「いつ記録するか」を定めているが、「その瞬間に気づく仕組み」が
なければ機能しない。特に、セッションが「壁打ちして仕様を固めよう」の
ような自由な形で始まった場合、決定確定の瞬間を知らせる外部トリガーが
存在せず、記録が自然と「一区切りついたところでまとめて」（バッチ生成）
に流れてしまうことが実際に確認された。

対策として、**会話の中で何かが決着したように見えた瞬間、AIが
「これをDecisionとして記録しておきましょうか？」と、その場で提案する**
運用を追加する。ここでAIが「これは決定になった」と自己判断して
勝手に記録することはしない（LLMの自己申告を信用しない原則に反する
ため）。あくまで提案に留め、実際に記録するかどうかは人間が判断する。
これにより、人間の記憶にも、AIの自己判断にも一方的に依存しない形で、
3.5の逐次記録を実質的に機能させる。

**3.7 ファイル数が増えたら、定期整合性チェックを儀式化する**
このプロジェクトでは、資産となるファイルが数個の段階では「気づいたら
その都度直す」で十分機能したが、ファイル数・参照関係が増えるほど、
気づかないまま放置される不整合（章番号のズレ、古い参照等）が増える。
ファイル数が一定以上（目安：5〜6ファイル超）になった時点で、
「気づいたら直す」から**「決まったタイミングで必ず全ファイルの
相互参照を確認する」**という儀式に格上げする。確認の際は`grep`等の
機械的な検索で章番号・IDの参照を突き合わせる（人間の目視だけに
頼らない。章番号の丸ごとの欠番は、目視では繰り返し見落とされた実例が
このプロジェクトである）。

## 4. スキーマの一元化とCGPの関係（誤解防止のための補足）

スキーマを1箇所にまとめることは、CGPの「対話人格を一つに保つ」という
原則と矛盾しない。一元化の対象は「モジュール間のデータの受け渡し方」であり、
「対話人格を複数にする」ことではないため、CGPの核（1章・2章）はそのまま維持される。
むしろスキーマの一元化は、モジュールの差し替えやすさというCGP本来の狙いを
強化する。

## 5. 全体の進め方（基本フロー）

1. **壁打ちから完成まで、原則1つのチャットで行う。** 製作者が検討事項について
   外部（別のAI・別のチャット）にアドバイスを求めることは可、ただし持ち帰った
   内容は必ずこのチャット内で検証・統合する（他人の壁打ちログをそのまま
   根拠として信用しない。3.1・グラウンディングの原則をここでも適用する）
2. **CGP方式（1〜4章）で作成を進める。** 人格は一つ・優先順位固定・
   スキーマ一元化・実行検証を通す
3. **Design Historyは3.5の通り、決定確定の都度・逐次記録する。**
   セクション完成時はレビュー・整理のみに使う

**補足（コンテキストウィンドウの制約への対応）**：チャットが長大化し
限界に達した場合は、新しいチャットへ移行してよい。その際は、直前までの
design_doc.md・各プロンプトファイル・スキーマファイル・Design History一式を
丸ごと引き継がせ、「1つのチャットで通す」という原則の趣旨（文脈の連続性・
検証可能性の担保）を実質的に保つこと。**その引き継ぎの標準フォーマットが
6章の引き継ぎパッケージ雛形である。**

## 6. 引き継ぎパッケージ雛形（LLM間・セッション間ハンドオフ）

CGP方式で育てたプロジェクトを、別のLLM・別のチャットセッションへ
引き継ぐための標準フォーマット。外部からの提案を検証・統合し、
このプロジェクト自身の規律（3.5〜3.7）を組み込む形で確定した。

### 構成（11ファイル）

```
00-readme.md          パッケージ全体の概要
01-project.md         プロジェクトの目的
02-philosophy.md      設計思想（開発者の好み・作業スタイルの記憶を含む）
03-invariants.md      絶対に変更してはいけない設計原則（IDで管理）
04-concepts.md        用語集（Not欄で混同を防ぐ）
05-decisions.md       採用された設計判断（現在の正として扱う）
06-rejected.md        却下された案
07-open-questions.md  未解決事項
08-current-state.md   現在の仕様を合成した唯一の正（design_doc.md相当）
09-next-task.md       次回セッション開始時に読むファイル
10-model-directive.md 引継ぎ先LLMへの指示
```

### CGPの規律との統合ポイント

- **IDは文書内の位置から独立させる**（`INV-001`、`DEC-004`等）。
  ファイルを再構成しても、既存の参照が壊れないようにするため
  （このプロジェクトで実際に起きた章番号ズレへの対策）
- **05-decisions.mdは3.5の逐次記録規律に従う。** 決定確定の直後に
  追記し、まとめて後から生成しない。各エントリには`Evidence`に加えて
  `Counter Evidence`（この決定に反する証拠）を必須項目として持つ
- **Confidenceは独自に判定せず、confirmation_sourceから導出する**：
  ```
  log_verified      → Confidence: High
  user_recollection → Confidence: Medium
  ai_reconstructed  → Confidence: Low
  ```
- **origin（3.6）を05-decisions.mdの各エントリにも適用する**：
  main_session／external_sessionを明記し、外部セッションでの
  偶発的な閃きも正直に記録できるようにする
- **Tension Axes（3.5）が該当する決定には、05-decisions.mdの
  該当エントリにも同欄を追加する**
- **08-current-state.mdは、05-decisions.mdの履歴を毎回読み解いて
  合成するのではなく、常に「今の状態だけ」を書き換える。**
  design_doc.mdと同じ役割分担（現在の状態＝1枚、経緯＝別ファイル）
- **10-model-directive.mdには、CGPプライマーの1〜2章（核となる原則）を
  要約として埋め込む。** 引継ぎ先LLMが本プライマー自体を読んでいない
  場合の保険とする
- **ファイル数が11と多いため、3.7の定期整合性チェックを必ず実施する**

雛形本体（実際にコピーして使える11ファイル）はGitHubリポジトリの
`handover/`に置く。`00-readme.md`は日本語・英語（`00-readme_en.md`）の
両方を用意する。それ以外のファイルはフィールド名（Decision、Status等）が
既に英語のため、説明文の翻訳は今のところ任意とする。

---

## 使い方

新しいアプリ設計のチャットを始める際、このファイルを冒頭で読み込ませ、
「このプライマーの原則に従って設計を進めてください」と伝えることを想定している。
5章の基本フローに従い、決定確定のたびにDesign Historyへの追記を促すこと。
セッションを跨いで引き継ぐ場合は、6章の引き継ぎパッケージ雛形
（`handover/`）を使用する。
