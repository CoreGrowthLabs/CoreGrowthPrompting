# CGP方式（公共性が高いケース向け）概要

個人用CGP（人格は一つ・耐ハルシネーション優先）をベースに、複数の
ステークホルダー間で価値観が対立しうる公共性の高い文脈向けに再設計したもの。
個人用CGPとは前提が異なるため、そのまま流用せず別方式として扱う。

---

## 1. 個人用CGPとの根本的な違い

個人用CGPが機能したのは、「本人が実際に言ったか」という**単一の事実**に
グラウンディングできたため。公共性が高い課題（プライバシーvs透明性、
効率vs包摂性、速度vs熟議、自動化vs人による監督など）は、グラウンディング
すべき単一の事実がそもそも存在しない。そのため、個人用CGPの核だった
「対話人格を一つに保つ」は捨て、**「対立を消さずに構造化して残す」**を
新しい核に据える。

## 2. 最初の振り分け：グラウンド可能か、対立含みか

すべての論点に対し、着手前に以下を判定する。

```
その論点は、単一の事実確認で決着するか？
        ↓
Yes → 従来のグラウンディング検証（事実ベース）を適用
No  → 「対立含み（contested）」タグを付け、Value-Tension Moduleへ回す
```

## 3. Value-Tension Module（対立検出モジュール）

- 既知の価値対立パターン（プライバシーvs透明性 等）に該当していないかを
  検出する専用モジュール。**解決はしない。検出のみ**
- AIの役割は「過去の類似事例・他分野の先例を横断的に検索し、対立の
  候補を広く拾う"スキャナー"」に限定する。AIが「これは対立だ」と
  自己判断して確定させることはしない（LLMの自己申告を信用しない原則の適用）
- 対立パターンのリストは、人間（それも単一の維持者ではなく、属性・立場の
  異なる複数人）が事前に作り、維持する。AIの提案はあくまで候補であり、
  そのままリストに追加しない

## 4. 非対称バイアス：迷ったら対立含みとして扱う

個人用CGPのJournal Moduleでは「過検出より見逃しを許容する」方針だったが、
公共版ではこれを反転させる。**見逃しのコスト（声の小さい当事者が不利益を
受け続ける）の方が、過検出のコスト（審議の手間が増える）より大きいため**、
対立の判定に迷った場合は対立含みとして扱う。

## 5. 「確認済み」の基準を変える

```
grounding_status（個人用）:
  confirmed = 本人の発言に根拠がある

deliberation_status（公共版）:
  reviewed = 実際に影響を受ける当事者グループの関与記録がある
  ai_drafted_only = AIが下書きしただけで、当事者の関与記録がない
```

`ai_drafted_only`のタグが付いた項目は、内部的にどれだけ一貫していても
「検証済み」の欄には入れない。

## 6. 「選ぶ」ロジックはコードに書かない

- 複数案（A案・B案）を根拠付きで並べて保持することは技術的に容易であり、
  積極的に行う
- しかし**「最終的にどちらを選ぶか」を決めるロジックをコードに書いた瞬間、
  そのロジック自体が新しい価値判断になる。** これは書かない
- 実際の選択は、コードの外側にある正当な人間の手続き（投票・審議会・
  当事者間の合意形成等）に委ねる
- 対立解決のルールが存在する場合も、それはコードが発明したものではなく、
  当事者があらかじめ合意したガバナンスルールを、コードが単に実行して
  いるだけである状態を保つ

```python
result = {
    "status": "contested",
    "options": [
        {"label": "プライバシー優先案", "rationale": "...", "supporting_data": {...}},
        {"label": "透明性優先案", "rationale": "...", "supporting_data": {...}}
    ],
    "resolution": None  # 意図的に空のまま。コードは選ばない
}
```

## 7. トレーサビリティ（監査ログ）

個人用CGPの「AIプロフィール（関係性の蓄積）」に相当するものとして、
**「どの決定が、どのステークホルダーの、いつの発言に由来するか」を
記録する監査ログ**を持つ。これがないと、AIが生成したもっともらしさが、
実際の代表性の代わりに使われてしまう。

## 8. 事後学習の記録は公開する

見逃していた対立が後から判明した事例は、個人用CGPのrejected_patternsと
同じ発想で記録し、次から同種の構造を見逃さないためのパターンとして
蓄積する。ただし公共版では、**この記録自体を公開し、外部の目でも
検証できるようにする**。

## 9. 外部からの異議申立てチャネル

内部のリスト・審議体制に載っていない対立を、外部の当事者・団体が
指摘できる窓口を用意する。これは技術ではなく制度設計の話だが、
Value-Tension Moduleの見落としを補う上で不可欠。

## 10. 成果物への明示

たたき台を出す時点で、**「これは内部的に一貫性のあるAI生成の下書きであり、
ステークホルダーによる検証を経ていない」という表示を、免責事項としてでは
なく、成果物のフォーマットそのものに埋め込む**。

---

## まとめ：技術で解決できる部分／できない部分の切り分け

| 技術（AI・プログラム）で担える部分 | 人間の体制でしか担えない部分 |
|---|---|
| 事実ベースの論点のグラウンディング | 対立の最終的な認定 |
| 対立パターンの候補を広く拾う（スキャナー） | 対立パターンリストの維持（複数人・多様な立場） |
| 複数案を根拠付きで並べて保持する | 実際にどちらを選ぶかの意思決定 |
| 決定の出所を記録する（監査ログ） | 異議申立てを受け止める窓口の運営 |
| AI下書きである旨をフォーマットに埋め込む | 記録を公開し外部検証を可能にする制度 |

公共性が高い文脈でのCGPは、個人用よりも「AIにさせないことを設計する」
比重が大きい。AIの役割を、最終判断者ではなく見落とし防止のための
補助（広く網を張るスキャナー）に限定し続けることが核心となる。
